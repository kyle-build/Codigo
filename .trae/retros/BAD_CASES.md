# BAD CASES

## 误将“首页开发文档”理解为“仓库级文档体系重构”

- 问题现象：用户需求是改造首页模块的开发文档页面为官网文档风格，但实现方向误跑偏到仓库级 README/Docs 体系建设，随后需要回退改动。
- 违规根因：未先定位需求指向的具体模块（`apps/client/src/modules/devDocument`），对“开发文档”语义范围做了过度扩展假设。
- 修正方案：后续涉及“文档/规范/页面”等多义词，先在代码内定位入口路由与模块目录，再开始实现；对不在目标模块内的改动默认禁止提交。

## 画布平移误触发“请先选择组件”提示

- 问题现象：未选中任何组件时，通过上下左右操作（方向键或模拟方向键的平移入口）移动画布，会弹出“请先选择组件/请先选中组件”，干扰画布平移体验。
- 违规根因：组件快捷键 hook 同时监听了方向键上/下，并在未选中组件时统一弹提示；与画布平移同按键监听冲突，导致本应静默的画布操作被组件快捷键误报。
- 修正方案：对“组件移动”类快捷键在未选中时静默退出（不弹提示），仅对 delete/copy 等确需选中组件的动作保留提示；避免不同域快捷键复用同一按键时产生误导性交互。

## 画布缩放句柄导致点击需“长按”才出现

- 问题现象：调整为四边/四角缩放句柄后，点击组件内容区域时经常无法立即选中（表现为需要长按/换位置点击才出现缩放句柄）。
- 违规根因：选中判断依赖 `event.target instanceof Element`；当用户点击文本节点时 `event.target` 可能是 Text（非 Element），导致“直击组件 wrapper”判断失败，从而不触发选中。
- 修正方案：统一将事件 target 归一化为 Element（Text 节点取 parentElement），再做 `.closest(".component-warpper")` 判断，确保点击文本也能立即选中并显示句柄。

## 误把“子页面问题”直接按切页处理，未先确认“页面集”模型

- 问题现象：用户反馈“新建子页面后原 tab 丢失并变成 overview”，初始处理先按“新建子页后不要切页”落代码，但未先确认真实需求，遗漏了“普通页面不能直接挂子页，需要先建页面集”的核心约束。
- 违规根因：在页面层级模型仍有歧义时先进入实现，默认沿用旧的路径推导树思路，导致修复范围过窄，只修了表象切页问题，没有先校准信息架构。
- 修正方案：后续遇到“页面/子页/分组/导航”这类结构性需求时，先确认对象模型（页面、页面集、路由关系、显示规则）再编码；若现有 schema 不支持，先评估最小数据模型变更点，再实施 UI 修复。

## 高仿模板扩容后未同步评估 `preset` 存储上限

- 问题现象：`admin-console-standard` 扩展为高仿多页面 Vuestic 模板后，服务端启动同步默认模板时报错 `Data too long for column 'preset'`，导致新模板无法写入数据库。
- 违规根因：只关注了模板预设与壳层 UI 改造，没有同步检查 `template.preset` 的数据库列类型；`simple-json` 在 MySQL 下实际落为 `TEXT`，容量不足以承载新的大体积模板 JSON。
- 修正方案：凡是显著增大 schema/preset/json 体积的改动，提交前必须同时核对对应实体字段的数据库映射与容量上限；大对象统一优先使用 `LONGTEXT` 或显式 JSON 列，不再默认沿用 `simple-json`。

## React 19 下直接使用 `react-quill` 导致属性面板崩溃

- 问题现象：点击模板中的富文本组件时，属性面板挂载 `react-quill` 直接抛出 `findDOMNode is not a function`，导致路由错误页接管界面。
- 违规根因：在 React 19 环境中继续直接挂载依赖 `findDOMNode` 的旧封装组件，没有先校验第三方编辑器与当前 React 主版本的兼容性。
- 修正方案：富文本属性面板改为直接基于 `Quill` 实例手动挂载，避开 `react-quill` 的不兼容生命周期；后续引入编辑器类依赖前，先验证其对当前 React 主版本的兼容声明与运行时行为。

## 页面发布版本快照超限导致 `PUT /api/pages/me` 返回 500

- 问题现象：编辑器点击发布时，`PUT /api/pages/me` 返回 500；前端控制台报 `AxiosError: Request failed with status code 500`，发布弹窗无法打开。
- 违规根因：`page_version.schema_data` 仍使用 `simple-json/TEXT` 存储整份发布快照；多页面后台 schema 扩容后，版本快照体积超过 MySQL `TEXT` 上限，插入版本记录时直接失败。
- 修正方案：大体积页面快照统一改为 `LONGTEXT` 并显式做 JSON 序列化；凡是发布/模板/同步链路引入更大 JSON 载荷时，提交前同步核对实体列类型与初始化 SQL 是否匹配。

## 预览页误插空白占位并重复挂载画布导致首屏看不到组件

- 问题现象：从编辑器进入预览后，首屏只看到大片空白区域，误以为页面组件丢失；实际内容被挤到下方，且预览画布还被重复渲染。
- 违规根因：预览页桌面壳层分支中先渲染了一个仅设置尺寸的空 `div`，后面又连续挂载 `ObservedPreviewCanvas` 和 `PreviewCanvas`，导致布局被空白占位顶开且画布初始化执行两次。
- 修正方案：预览内容容器内只保留一次 `ObservedPreviewCanvas` 渲染，移除空白占位节点；后续排查“页面空白”类问题时，先检查容器结构和重复挂载，再判断 schema 与运行时数据链路。

## Vuestic 模板节点缺失 `id` 导致发布插入组件失败

- 问题现象：普通手搭页面可以发布，但 `Vuestic Admin 高仿模板` 发布时依然报 500；继续排查后确认服务端插入 `component` 表时报 `Field 'node_id' doesn't have a default value`。
- 违规根因：发布链路默认假设客户端上传的 schema 组件树都带 `id`，但模板型数据可能存在缺失节点标识的情况；服务端没有在入库前做结构规范化，导致 `node_id` 直接写入空值。
- 修正方案：发布服务在解析 schema 时统一递归补齐组件节点、页面、页面组的缺失 `id`，并把补齐后的 schema 回写到版本快照；后续所有模板/导入型 schema 入库前都必须先做规范化。
